# 実装計画：施設側へのワーカー本人電話番号の表示（緊急時のみ）

作成日: 2026-07-01
ステータス: 仕様確定・実装未着手
関連メモリ: worker-phone-facility-disclosure / facility-data-worker-exposure-paths / notification-template-db-sync-gap

---

## 1. 目的・背景

施設側からワーカー本人の電話番号が確認できず、緊急時（出勤時刻超過で連絡がない等）に連絡手段がない。
緊急連絡先（関係者）だけでなく、**ワーカー本人の電話番号**を施設に表示する。
ただし常用連絡を防ぐため「緊急時のみ利用」の注意書きを併記する（カイテク準拠）。

## 2. 確定要件（クライアント回答 2026-07-01）

| # | 項目 | 確定内容 |
|---|---|---|
| ① 表示タイミング | カイテク仕様＝**応募段階から表示**。現行の緊急連絡先モーダルは応募段階から施設が開けるため、モーダルに本人電話を追加するだけで満たす |
| ② 注意書き | 「緊急時のみご利用ください」＋3ケース。カイテクの「ご負担になります」表現は**削除** |
| ③ ワーカー告知 | 本人電話が施設に緊急時用として共有される旨を、規約・登録/プロフィール画面に明記 |

## 3. スコープ

### 対象（IN）
- 施設側の緊急連絡先モーダル（`app/admin/workers/[id]/page.tsx`）へ本人電話＋注意書きを追加
- ワーカー告知（登録画面・プロフィール編集画面のインライン文言／プライバシーポリシー本文）

### 対象外（OUT・今回スコープ外）
- 勤怠一覧・応募一覧など他画面への本人電話表示（モーダルのみ）
- 濫用防止のためのアクセスログ／通話課金の仕組み
- 別ルート `app/admin/workers/[id]/emergency-contacts/page.tsx`（現在未リンクのため）

## 4. 現状（調査結果）

- 本人電話 `User.phone_number` は `getWorkerDetail`（`src/lib/actions/review-admin.ts:180` → `phone: user.phone_number`）で**既に取得済み・UI未表示**。
  → **DB変更・スキーマ変更・追加取得は不要。デプロイのみで反映。**
- 表示コンポーネントは `app/admin/workers/[id]/page.tsx` のインラインモーダル（767〜834行）。ボタンは622行。
- ワーカー側プライバシーポリシーはDB管理（`LegalDocument` テーブル、`getLegalDocument('PRIVACY','WORKER')`）。CMSは `app/system-admin/content/legal/page.tsx`。既存環境はseed変更では反映されず**CMSからの手動更新が必要**（[[notification-template-db-sync-gap]] と同型）。

## 5. 実装タスク

### タスクA：施設モーダルUI改修（本体・最重要）
**ファイル**: `app/admin/workers/[id]/page.tsx`（緊急連絡先モーダル 750〜837行）

1. モーダルコンテンツ（767行〜）の**先頭に「本人への緊急連絡先」セクションを新設**
   - `worker.phone` を表示（`tel:` リンク、現行の関係者電話と同じ挙動）
   - 未登録時フォールバック「登録なし」（`phone_number` は必須項目のため通常は存在）
2. その直下に**注意書きボックス**（amber/red系）を追加：
   ```
   緊急時のみご利用ください。下記以外の目的でのご連絡はお控えください。
   ・出勤時刻を過ぎても本人からご連絡がない場合
   ・面談の日程調整、合否連絡など採用に関して早急な連絡が必要な場合
   ・やむを得ない事情によりお仕事をキャンセルする場合
   ```
3. 既存の緊急連絡先ブロックを**「関係者への緊急連絡先」の見出し**に変更し、説明文を追記：
   「お仕事中に本人に重大な事故などがあった際の緊急連絡先としてご利用ください。」
4. アイコン import 追加（警告用）: `app/admin/workers/[id]/page.tsx:6` の lucide-react import に `AlertTriangle` を追加（本人電話は既存 `Phone` を流用）。

DB変更なし・型変更なし（`worker.phone` は `useAdminWorkerDetail` の型に既存）。

### タスクB：ワーカー告知（インライン文言）
デプロイで即反映されるコード側の最小告知。

1. **登録画面**: `app/register/worker/page.tsx` 電話番号入力（887行付近）の近くに注記：
   「登録した電話番号は、緊急時に限り応募先の施設へ共有されます。」
2. **プロフィール編集**: `app/mypage/profile/ProfileEditClient.tsx` 本人電話フィールド（1268行付近）の近くに同旨の注記。

※既存の登録フローには利用規約・プライバシーポリシーへの同意チェックが既にあるため、**新規の必須同意チェックボックスは追加しない**（プライバシーポリシー本文の更新で同意根拠を担保）。

### タスクC：プライバシーポリシー本文の更新（内容・手動反映）
1. `prisma/seed-legal-documents.ts`（privacyContent, 486行付近「3. 個人情報を収集・利用する目的」）に、施設への連絡先共有（緊急時利用）の記載を追記 → **新規環境用**。
2. **既存の本番・ステージングは反映されない**ため、`/system-admin/content/legal` のCMSから**手動で本文更新**（[[notification-template-db-sync-gap]] と同じ運用注意）。文言はクライアント/先方の最終確認を推奨。

> 登録した電話番号は、緊急時に限り応募先の施設へ共有されます。

## 9. リスク・注意

- **露出方向の確認**: 今回は「本人電話を*施設*に見せる」方向。メモリ [[facility-data-worker-exposure-paths]] の PR #628（施設代表電話を*ワーカー*に隠す）とは**逆方向で別件・矛盾なし**。
- **プライバシーポリシーのDB同期ギャップ**: seed変更だけでは既存環境に反映されない。CMS手動更新を忘れないこと。
- **表示範囲を広げない**: 将来「勤怠一覧にも出したい」となった場合は露出経路が増えるため、その時点で経路ごとの `select`/型/表示を一括対応する（今回は対象外）。
