1. 概要

ワーカーが勤務実績の前払いを、給与日を待たず1円単位でいつでも引き出せる機能。GMOあおぞらネット銀行APIで送金する。

  • 前払い成立条件: 勤怠完了（CHECKED_OUT）＋ 本人のレビュー提出。
  • 前払い率: 既定9割（社保対象見込みなどは7割等、個人ポリシーで設定）。
  • 締め: 月末23:59。1日に当月残額＋プールを給与口座へ振替（スイープ本体は未実装）。
  • 公開状態: feature flag OFF + GMO dummy のため本番ユーザーには非表示・無影響。
公開スイッチは環境変数 NEXT_PUBLIC_FEATURE_HIBARAI=true（ビルド時焼き込み）。これをONにして再デプロイするまで、UIは出ず Server Action / cron / API も全て早期returnで無効。

2. アーキテクチャ（金銭整合の核）

残高は「台帳の合計」

残高 = SUM(point_ledger_entries.delta) という不変条件で管理。サマリ point_balances は台帳と常に一致するよう、台帳追記と残高更新を同一トランザクションで行う。

台帳の種類(kind)意味delta
ATTENDANCE_CONFIRMED勤怠確定でチャージ（前払い率を保存）+前払い額
WITHDRAWAL_RESERVED出金申請で引当−申請額
WITHDRAWAL_COMPLETED送金完了（状態遷移のみ）0
WITHDRAWAL_REVERTED送金失敗で返却+申請額
MANUAL_ADJUSTMENT勤怠修正後の差額調整 等差額

出金フロー

レビュー提出 ──▶ reconcile（チャージ: +残高, ATTENDANCE_CONFIRMED, 率を保存） │ ▼ 受取申請 ──▶ 残高ロック(FOR UPDATE)・引当(−残高, WITHDRAWAL_RESERVED)・口座snapshot保存 │ ＋ 緊急停止/上限/負残高/プログラム(HIBARAIのみ) を検査 ▼ cron: submit-pending（毎分）──▶ 初回のみ口座再検査 ──▶ GMO送金 ──▶ applyNo保存(PROCESSING) │ ＋ transfer_attempts に試行記録 ▼ cron: poll-transfer-status（5分毎）──▶ GMO状況照会 20:完了→COMPLETED / 40等:失敗→返却(+残高) / 26:組戻不成立→残高戻さず要調査

二重送金・過払いの主な防御

  • 残高ロック SELECT … FOR UPDATE ＋ Serializable トランザクション（並行二重引当を防止）
  • 不足/負残高は InsufficientBalanceError / NegativeBalanceError で停止
  • GMO送信は Idempotency-Key（再送も同一キー）＋ claim の原子的 updateMany で1回のみ
  • 台帳の idempotency_key・gmo_apply_no は UNIQUE、1勤怠1チャージの部分UNIQUE
  • Webhook は message_id UNIQUE で重複処理防止、署名はHMAC-SHA256検証

Architect・Codex レビューで挙がった P0 を、全てテスト先行（TDD）＋ Codex 厳格レビューで実装。

P0-1 精算月出金に settlement_month を付与。月末スイープ/給与控除の集計キー。JST月定義をチャージ/出金で統一（月跨ぎバグ修正）settlement_month完了

4. 金銭リスクレビュー結果

Claude 3エージェント（二重払い / 過払い / 日付跨ぎ）＋ Codex の独立レビューを実施。

CRITICAL（発見・修正済） GMO組戻不成立(status=26)で残高を誤復元
組戻し(recall)が失敗し資金が受取人側に残っているのに、失敗扱いで残高を戻していた → 再出金で運営者が二重損失。
→ 修正: 26 を残高復元対象から除外（isBalanceRestorableFailureStatus）。送金済み扱い＋ERROR監査で手動調査へ。ユニットテストで固定。
※ Codex が発見。Claude 3エージェントは見落とし。多層レビューの価値。

上記以外は、3エージェント＋Codex とも「運営者が損する確定的な経路は無い」で一致。

ユーザー懸念への回答

懸念結論
残高が無いのに支払えてしまう起きない。FOR UPDATE＋Serializableで残高検査、不足/負残高で停止
二重払い起きない（多層防御）。残るのはGMO Idempotency-Key仕様への依存1点（実機確認要）
日付跨ぎで支払ったのに残高が減らない起きない。引当(−残高)と台帳追記は同一原子トランザクション。JST/@db.Date も正しく処理

5. cron 構成

cron は migration ではなく vercel.json の crons で設定（Vercel Cron Jobs）。今回の機能は3つ使用。フラグOFF時は404で無害。認証は CRON_SECRET（Vercel環境変数）。
cronスケジュール役割
hibarai-submit-pending* * * * *（毎分）PENDINGの出金をGMOへ送信
hibarai-poll-transfer-status*/5 * * * *（5分毎）GMO振込状況を照会し完了/失敗を確定
hibarai-refresh-oauth-token0 * * * *（毎時）GMO OAuthトークン更新
hibarai-gmo-balance-alert*/30 * * * *（30分毎）送金元残高が閾値割れでSystem Admin全員へ通知

6. 失敗系の運用（残高不足・口座誤り・通知）

送金元（GMO）残高不足のとき

残高不足での出金は FAILED にしない。「GMO到達が不明」として PROCESSING を維持し、残高は予約したまま（ワーカー画面は「失敗」ではなく「処理中（未振込）」）。hibarai-submit-pending（毎分・各依頼は約2分間隔）が 自動で再送するため、運営は送金元口座へ入金するだけでよく、入金後にキューが自動的に流れる。管理者による手動の再振込は不要。
  • 資金ゲート: hibarai-submit-pending は送金元の引き出し可能残高で送れる依頼だけを送信し、賄えない分は未送金のまま据え置き（GMOへの空振りリクエストを削減）。GMO未接続/dummyで残高取得不可のときはゲートせず従来どおり全件試行。純粋関数 planFundedSubmissions でTDD。
  • ダッシュボード可視化: 「未送金（処理待ち・入金待ち）」の件数/金額を表示（status ∈ {PENDING,PROCESSING} かつ gmo_apply_no 未発行の合計）。残高不足だとここが増える。
  • 督促: hibarai-gmo-balance-alert が枯渇前にSystem Admin全員へ通知（注意/警告/危険の3段階）。

口座誤りのとき（再振込＝再処理は作らない）

GMOに事前の口座有効性/名義照合API は無い（公開資料に記載なし・要GMO確認）。口座の正否は実際に振り込むまで保証されない。名義は登録時に自動カナ化のみのため、名義不一致が起こり得る。
  • 口座誤りは多くの場合 GMO 受付後の振込失敗（status 40「口座を確認できませんでした」等）として返り、hibarai-poll-transfer-status が残高を戻して FAILED（＝実質キャンセル、自動再送はしない）。
  • FAILED は日次/月次上限のカウントから除外されるため、ワーカーは口座を直して再依頼すればよい（管理者の再振込機能は不要と判断）。
  • 組戻不成立(26) は資金が受取人側に残るため COMPLETED 維持・残高戻さず。通知せず、A3エラーキュー＋手動残高調整で個別対応。

失敗通知＋初回口座確認

  • 失敗通知 WORKER_WITHDRAWAL_FAILED（メール＋プッシュ）: status=FAILED のときだけ送信（26/COMPLETED・PROCESSING在中は通知しない）。口座確認・再申請URL・サポートコードを含む。完全防御（通知失敗が出金フローを止めない）。テンプレ行はDB（seed-notifications.ts/管理UI）に登録が必要。
  • 初回口座確認トースト: 口座登録済み＆過去出金0件の初回受け取り画面で、口座（特に名義カナ）の確認を促す非強制トーストを表示。

8. 公開前チェックリスト

公開前 必須（これが済むまで実送金を有効化しない）

  • 4. 監視（PROCESSING滞留 / UNKNOWN・RECALL_FAILED監査頻度 / GMO明細日次突合）
