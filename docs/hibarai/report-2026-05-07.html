エグゼクティブサマリ

タスタス内で、ワーカーが 確定済み報酬を「ポイント残高」として保有 し、GMO あおぞら API による 24/365 即時振込で前払いを受けられる仕組み を構築する。既存の「キャリ払い（キャリバライ）」に近いが、別システムとして新規構築する方針。

本 MTG（2026-05-07）で 主要な仕様骨子は確定 した。残課題は ① GMO API の細部（口座名義照合の API 化可否、Webhook 範囲、組戻し）② キャリバライ仕様の再現範囲 ③ 利用規約変更のタイミング ④ 個人別レート設定の運用 UI／UX、の 4 領域に集約される。

1MTG#1 で確定した仕様骨子

議事録に基づき、ステータスが ✓ 確定 となった項目のみ列挙。

1.1 機能の前提

前払い対象確定済み報酬のみ（業務委託報酬の早期受領）
前払い成立条件本人の勤怠完了 ＋ 本人レビューの提出（施設側の承認は待たない／振込を止めない）
既存「キャリバライ」近いが別システムとして構築。仕様再現範囲は要共有
送金経路GMO あおぞら API（24/365 即時振込）

1.2 残高・申請

ポイント定義確定報酬は自動的にポイント残高へ加算（1pt = 1円のウォレット型）
前払い率（基本）勤務実績の 9 割をプール（前払い可能上限）
前払い率（例外）労働者性が否認できない場合の留意等により 7 割等、ワーカー別に変動。当面は個別設定方式
引き出し単位1 円単位
交通費前払い対象金額に 含めて チャージ可
申請可能時間帯24/365 申請可。完了は銀行都合に準じる（GW 等は停止可能性あり）

1.3 手数料

負担者ワーカー本人負担
体系1 回ごとの固定額（100〜200 円程度）を予定（具体額は未確定）
控除方法チャージ済み残高から控除
設計意図まとめて引き出した方が得になる料率設計

1.4 セキュリティ・不正対策

口座変更時メール／SMS 認証を挟む（ロック方式ではなく認証方式）。本人名義一致は必須
緊急停止システム管理者による 全前払い一括停止スイッチ
基本制約勤怠に紐づいた金額以上は出さない（最小限の制約で十分という方針）
過剰対策原則しない（運用が複雑化するため）

1.5 ログ・履歴

ログの作業日勤務開始時の日時（「いついつ分の仕事がチャージされたか」を記録）
履歴保持5 年保持を想定（振込関連の法定保存期間に準拠／最終未確定）
施設側 UI前払い実行情報は 施設には表示しない

2月末締め・支払サイクル

議事録 第 7 項に基づく。残額（プール 1 割分含む）は各案件の既存報酬支払日に受取口座へ振込。

勤務中
ワーカーが勤怠完了 ＋ 本人レビュー提出 → 確定額（交通費込み）がポイント残高に加算
月内随時
ワーカーが前払い申請（24/365、1 円単位、上限 = 確定額 × 個人設定率） → 手数料控除 → GMO 即時振込
当月末日
23:59
前払い申請 締切（以降、引き出し条件が切替）
翌月 1 日
月次精算処理：残額（プール分の 1 割含む）を 各案件の既存報酬支払日に向けた受取口座への振込手続き
既存支払日
既存の報酬支払日として 受取口座へ振込完了 → 残高ゼロでクリア
運用ポイント: 報酬支払日は全社統一せず、案件ごとの既存支払日を維持する設計。月初 1 日は精算バッチの起動日。月末締めにより、ポイント残高の自動失効ルール（B2）も同時に成立する。

3個人別 前払い率の設定方針

議事録 第 3・4・15 項より。基本 9 割＋例外 7 割の個人別運用とする。

観点内容
基本率9 割（全ワーカー初期値）
例外率労働者性が否認できない場合の留意が必要な場合は 7 割等
算出要素勤務形態 / 勤務時間 / 報酬単価 / 契約条件 / 労働者性が否認できない場合の留意
設定方式当面は個別手動設定可能な形（将来は勤務形態からの自動算出を検討）
上限金額将来的に個人ごとの上限金額（1 回／1 日／1 月）も設定可能とする

運用上の論点

  • 9 割→ 7 割への切替は、施設とワーカーの合意により決定後、施設側で設定する案（未決定）
  • キャリバライの設定仕様をどこまで再現するか
  • 労働者性が否認できない場合の留意点をシステムで検知するか、運用判断か

4GMO あおぞら API 調査結果

公式公開情報からの所見。詳細は GMO 開発者ポータルおよび直接ヒアリングで確定。

4.1 API ラインアップ（無償・標準 API）

標準 API 28 種類（無償）＋ オプション API 7 種類（有償）の計 35 種類。本件で利用見込みのものは以下：

API用途本件での利用見込み
振込依頼仕向け振込指示（仕様の中核）必須
振込状況照会振込履歴・処理状況の照会必須
振込依頼結果照会依頼／取消の処理状態の照会必須
振込手数料事前照会手数料を申請前に取得推奨
振込取消依頼振込予約・申請の取消推奨
残高照会自社運用口座の残高監視推奨
振込入金口座発行バーチャル口座（個別 ID）の発行将来検討
通知配信制御 (Webhook)イベント通知の配信制御推奨
振込入金口座 入金明細通知被仕向（着金）の Webhook将来検討

4.2 技術仕様

認可OAuth 2.0 ＋ OpenID Connect
プロトコルHTTPS（TLS 1.2 以上）
データ形式JSON
アーキテクチャREST
SDKGo / PHP（GitHub に公式 SDK 公開）
テスト環境sunabar（API 実験場、無料、本番接続前にフル検証可能）

4.3 振込手数料・限度額

項目通常会員振込料金とくとく会員（月額500円）
同行宛無料無料
他行宛143 円（税込）/ 件129 円（税込）/ 件
組戻手数料880 円（税込）/ 件
振込限度額（自動）0 円〜5 億円（初期値 500 万円）
振込限度額（審査）5 億円超〜 99 億円
⚠️ 要注意: 公式ページに 「2026 年 5 月 10 日より変更予定」 の注記あり。手数料額が改定される可能性が高く、ワーカー負担額（100〜200 円想定）の妥当性に直接影響する。GMO 担当者へ早期に確認を推奨。

4.4 24/365 と例外時間

  • 同行内振込：24/365（システムメンテナンス時を除く）
  • 他行宛振込：原則 24/365。毎週土曜 23:52〜00:10 / 日曜 23:52〜00:10 は統合 ATM センター計画停止
  • 送金先金融機関によっては 翌営業日着金 となるケースあり
  • UI 表示推奨：「申請はいつでも可／完了は銀行に準じる」（議事録 第 6 項）

4.5 振込先口座確認（名義照合）

対応金融機関全銀モアタイムシステム参加金融機関（メガバンク・地銀・信金等の多くを含む。詳細は同システム参加機関一覧で確認）
利用時間24/365（毎週土曜 23:52〜00:10 / 日曜 23:52〜00:10 は停止）
API 化の可否要問い合わせ 公式公開情報ではネットバンキング画面での利用のみ言及。API 提供されているかは GMO 担当者への確認が必要
名義照合精度要問い合わせ 完全一致／近似マッチの仕様は公開資料からは不明
重要な未確認事項: 議事録 第 13 項で「口座名義は本人名義一致が重要」とされており、本機能は受取人名義の自動検証に依存する可能性が高い。API 経由で名義照合が利用可能かどうかは GMO へ最優先で確認すべき項目。利用不可の場合は、ワーカー登録時の本人名義確認を強化する代替設計が必要。

4.6 Webhook（イベント通知）

  • 「振込入金口座 入金明細通知」（被仕向、すなわち着金）の Webhook は提供されている
  • 仕向（自社からの振込）の 振込結果通知 Webhook は公開資料からは確認できない
  • 提供されていない場合は ポーリング（振込状況照会 API）方式 で結果を取得する設計となる

5残課題・要確認事項

MTG#1 で「未確定」または新出した項目。次回 MTG ／ クライアント側調査で順次クローズ。

5.1 GMO 側へ確認すべき事項（最優先）

  • 口座名義照合の API 化可否（最優先）— 本人名義一致の自動検証ができるか
  • 仕向振込の結果 Webhook の有無 — 失敗時の即時検知・組戻し対応に直結
  • 組戻し処理の API 化可否 — 振込失敗時のリカバリー（組戻手数料 880 円が発生する想定）
  • 2026/5/10 の手数料改定の詳細 — ワーカー負担額（100〜200 円）の妥当性に影響
  • レート制限・並列処理性能 — ピーク時の同時前払い申請への耐性
  • エラーコード体系の全量 — エラー時 UX（議事録 第 12 項）の詳細設計に必要

5.2 キャリバライに関する確認すべき事項

  • キャリバライの仕様再現範囲 — 率設定 / UI / 運用フロー
  • 個人別の前払い率の設定 UI と運用（system-admin での操作主体・承認フロー）
  • 労働者性が否認できない場合の留意点の自動化要否（自動検出か、運用判断か）
  • 利用規約変更の要否と時期（第 8 条「報酬等支払の立替え」の改定 or 前払いサービス利用規約の新設）
  • 残課題項目（D3／A2／A3／A5／C4／C6／F1／F4／I1〜4／J1〜3／K1-2／K4／L1-5／N2／O1-2／P1-4）の方針

6リスクと留意点

仕様確定・実装・ローンチに向けた主要リスク。

R1: 手数料改定（2026/5/10）

他行宛振込手数料が改定される可能性。ワーカー負担 100〜200 円との整合性が崩れると、料金体系の再設計が必要。

R2: 名義照合 API 不可

口座名義の自動照合が API 化されていない場合、不正・誤振込リスクが上昇。代替策として、登録時の本人名義確認の強化または手動レビュー導入が必要。

R3: 振込結果 Webhook 不在

仕向振込の結果通知 Webhook が未提供の場合、ポーリング方式となり、ワーカーへの完了通知 UX に遅延が生じる。

R4: 既存 BankAccount 平文保存

現状、口座情報は平文保存。前払い機能と同時に 暗号化対応（at rest） を必須化すべき。

R5: 勤怠修正による残高ズレ

確定後の勤怠修正で前払い済額を超える減額が発生した場合の運用フローが未定。マイナス残高許容 / 追加請求 / 別途精算のいずれを採るか要決定。

R6: 一括ローンチ方針

「完全仕様確定後の一括リリース」方針のため、全項目クローズまでローンチ不可。残課題が多く残ると遅延に直結。

7推奨アクション・次のステップ

3 つのトラックを並行して走らせる構成を推奨。

トラック A：GMO 接続準備（申込済み・進行中）

  • 口座名義照合 API、振込結果 Webhook、組戻し API、2026/5/10 手数料改定の詳細を担当者に確認
  • sunabar（テスト環境）への接続申請、SDK 動作確認（Go／PHP）

トラック B：仕様クローズ（次回 MTG 群）

  • キャリバライの仕様取得 → 再現範囲の決定セッション
  • 個人別レート設定の運用 UI／UX 詳細化
  • 勤怠修正による残高調整方針の確定
  • エラー時 UX（通知文言・口座修正導線・組戻しフロー）の詳細化
  • 通知媒体（アプリ内 / メール / SMS）の選定
  • 残課題（5.2、5.3 の各項目）を順次クローズ

トラック C：規約・法務

  • 求職者向け利用規約 第 8 条「報酬等支払の立替え」の改定要否レビュー
  • 前払いサービス利用規約の新設要否レビュー
  • 規約改定時期と告知計画の決定
