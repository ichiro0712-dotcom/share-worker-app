## 1. Bottom Line

総合成熟度は **55%**。UI/モック/画面分解はかなり進んでおり、社内レビュー用プロトタイプとしては有効。一方、**Prisma設計・Server Actions・GMO連携の本実装へ進むには NO-GO**。

理由：支払日、算定母数、勤怠確定、報酬控除、組戻し、キャリ払い排他、口座変更セキュリティが、成果物間でまだ未確定または矛盾しているため。特に **「特定日固定の支払い」前提と、案件ごとに異なる既存の報酬支払日を維持する方針が衝突**している。

## 6. DB設計前に確定すべきこと

| 項目 | 必須決定 |
|---|---|
| 支払スキーム | 報酬前払い / 立替 / 貸付 / 外部代行 のどれか |
| 支払日 | 案件ごとの既存報酬支払日を維持し、日払い残額をそこへ乗せる設計 |
| 算定式 | 報酬総額/控除後、交通費、深夜残業相当、税・契約上の控除、手数料、上限を含む式 |
| 勤怠適格条件 | `Application` × `Attendance` × `ReviewStatus` のどの組み合わせで解放 |
| 台帳 | `Withdrawal` だけでなく `EligibilitySnapshot`, `TransferAttempt`, `LedgerEntry`, `ContractFeeDeduction` が必要 |
| 口座 | 既存 `BankAccount` には status/cooldown/version/verifiedAt がない |
| 返金/組戻し | reversal、refund、negative balance、報酬控除不能時の債権管理 |
| 監査 | append-only、ハッシュ検証、閲覧ログ |
| 排他制約 | 同一勤怠/同一報酬期間/キャリ払い/新日払い/報酬本払いの一意制約 |

## 7. API設計前に確定すべきこと

| 領域 | 必須インターフェース |
|---|---|
| 残高 | `getMoneyHome`, `getBreakdown` はサーバー計算値のみ返す |
| 申請 | `createWithdrawalSnapshot` → `confirmWithdrawal` の2段階。冪等キー必須 |
| レビュー | 既存 `submitReview` 後に日払い解放処理を同期/非同期どちらで走らせるか |
| 口座 | 登録/変更/OTP/クールダウン/口座バージョン固定 |
| GMO | 送金依頼/Webhook/失敗分類/再処理/組戻し |
| 報酬 | `advance deductions by payment period` 出力と報酬締め済みロック |
| 管理者 | 緊急停止/解除申請/二者承認/設定変更申請 |
| エラー | ユーザー向け理由分類と管理者向け内部コードの対応表 |

## 8. 法務・経理・契約との合意必要事項

| 領域 | 合意事項 |
|---|---|
| **法務必須** | 手数料を本人負担にする根拠、控除同意、利用規約/業務委託契約条項、貸金業該当性、業務委託契約書の改訂 |
| **経理必須** | 前払い仕訳、手数料収益/実費、組戻し、報酬控除、未回収金 |
| **契約必須** | 報酬支払日、報酬明細/支払調書表示、委託料の控除条項、源泉徴収（看護師・介護士の業務委託の特定業務該当性）、交通費の扱い |
| **セキュリティ必須** | 口座変更後停止、MFA、二者承認、監査ログ、GMO鍵管理 |

## 9. 次のMTG議題候補（10項目）

1. 案件ごとの既存報酬支払日を維持し、日払い残額をどう紐づけるか
2. 前払いの法的・会計的性質を何として扱うか
3. 手数料は報酬控除か、別建て利用料か、会社負担か
4. 前払い可能額の算定式を1本化する
5. 勤怠完了、レビュー完了、施設確認、勤怠修正の状態表をDB名で確定する
6. 施設修正・取消後の過払い回収ルールを決める
7. 月跨ぎ夜勤と月末23:59前後の精算月を決める
8. キャリ払いとの併用/移行期間の排他ルールを決める
9. GMO方式、Webhook、名義照合、組戻しの責任分界を決める
10. Prismaに入れる台帳モデルと報酬連携データ契約を合意する
