# 日払い機能 セキュリティ・コンプライアンスレビュー

**レビュー日**: 2026-05-27
**レビュアー**: Codex (Security Analyst)
**対象**: docs/hibarai/2026-5-7.md（日払い機能）

---

## 1. Threat Summary

全体評価は **CRITICAL**。理由は、勤怠・レビュー・口座・GMO送金APIのどれか1点が破られるだけで、不可逆な金銭移動が発生するため。

**主要資産**: 勤務実績、前払い可能額、銀行口座情報、GMO API鍵、送金台帳、監査ログ、管理者停止権限
**脅威主体**: 乗っ取り済みワーカー、悪意あるワーカー、施設側担当者、内部管理者、API鍵窃取者、Webhook偽装者

**OWASP上の中心**: A01 Broken Access Control / A02 Cryptographic Failures / A04 Insecure Design / A07 Identification and Authentication Failures / A09 Logging and Monitoring Failures
**API固有**: BOLA/IDOR、レート制限不足、Webhook検証不足

## 2. Critical Vulnerabilities（CRITICAL）

### CRITICAL-1: 口座変更後すぐ出金できる設計
- **Impact**: セッション奪取・メール/SMS乗っ取り時に、攻撃者口座へ即時送金
- **Remediation**:
  - 口座登録/変更は「現在パスワード再入力 + SMS OTP + メール通知」
  - 変更後24時間は出金停止、新規端末/新IPなら72時間または少額制限
  - 変更前後の口座はマスクして監査記録
  - 既存セッションは全失効

### CRITICAL-2: 前払い可能額をクライアント値・可変設定に依存
- **Impact**: 9割/7割、勤怠金額、月跨ぎ、取消勤怠の改竄で不正出金
- **Remediation**:
  - サーバー側のみで、承認済み勤怠・レビュー・支払期間・既払額・控除見込を元に整数円で再計算
  - `rate` はDBポリシーで有効期間管理、変更は二者承認
  - 送金前にトランザクションロックと一意な `withdrawal_idempotency_key` を必須化

### CRITICAL-3: GMO API鍵・Webhook検証の不備
- **Impact**: API鍵流出で大量送金、偽Webhookで支払済み偽装
- **Remediation**:
  - Vercel環境変数だけで完結させず、可能ならKMS/Secrets Manager
  - 鍵ローテ、送金権限用サブアカウント、日次/件数上限、IP制限
  - GMOあおぞらWebhookは `x-webhook-signature` をHS256で検証
  - User-Agentだけを信用しない
  - TLS1.2以上、OAuth2/OIDC、Webhook署名項目

### CRITICAL-4: 管理者の緊急停止/解除が単独権限
- **Impact**: 悪意ある管理者または管理者乗っ取りで、全件送金・停止解除・証跡破壊
- **Remediation**:
  - 管理者MFAはWebAuthn/TOTP必須
  - 停止は単独可、**解除・上限変更・手動送金は二者承認**
  - 全操作をappend-only監査ログへ記録

## 3. High-Risk Issues（HIGH）

- **パスワードリセット悪用**: リセット完了時は全セッション失効、24時間は口座変更/高額出金不可。リセットトークンはハッシュ保存、15分TTL、単回使用
- **IDOR/BOLA**: `workerId` や `attendanceId` をリクエストから信用しない。セッションから主体を導出し、DB側でもテナント/本人スコープを強制
- **連続申請・二重送金**: 同一勤怠・同一支払期間・同一ワーカーに対する残高計算は直列化。GMO送金リトライは必ず冪等キーを使う
- **口座名義不一致**: カナ正規化、法人/個人名義禁止、本人名義一致が取れない場合は初回送金不可。失敗理由はユーザーへ詳細開示しない
- **口座番号・名義の平文保存**: アプリケーション層のエンベロープ暗号化、検索用には正規化値のHMACのみ保存。UI/API/ログはマスク
- **監査ログ改竄**: 送金系はUPDATE/DELETE不可のappend-only。日次でWORM相当の外部ストレージへハッシュ付きエクスポート

## 4. Medium/Low

**MEDIUM**: SMS単独認証、CSRF/CORS不備、Prisma外のraw SQL、XSSによる管理画面乗っ取り、サポート経由のソーシャルエンジニアリング、GMO障害時の再送制御不足
**LOW**: メールアドレス列挙、過剰ログ保持、退会後の不要データ残存、管理画面の権限名混同

## 5. 法令遵守チェック

### 労基法24条（賃金支払い5原則）
- 通貨払い・直接払い・全額払い・毎月1回以上・一定期日払い
- 本機能は「**既に発生した賃金の前払い**」であり、労働者が返済義務を負わず、通常支払日に残額を精算する設計なら、**貸付ではなく賃金支払いの一形態**として整理しやすい
- ただし手数料を賃金から控除するなら、労基法24条の全額払い例外として**労使協定と個別同意が必要**
- 最も安全なのは事業者/施設負担、次善が明確な任意利用料として別建て徴収

### 貸金業該当性
- 金融庁/経産省の2018年グレーゾーン回答: 勤怠実績に応じた賃金相当額を給与口座に振り込む前払いは、事実関係次第で貸金業に該当しない余地あり
- **給与ファクタリング型**（賃金債権買取・労働者からの回収・高額手数料）は貸金業該当リスクが高い
- 金融庁は給与ファクタリングを貸金業に該当し得るものとして注意喚起

### 給与デジタル払いとの区別
- 銀行口座への振込なら通常の口座振込
- 資金移動業者口座へ賃金を払う場合は、厚労大臣指定業者、労使協定、本人同意が必要

### その他
- 古物営業法は通常不適用
- 前払式支払手段も、ポイント/残高を発行して決済に使わせない限り通常不適用
- **保存期間は5年設計で妥当**。労働関係書類は法令上5年へ延長、経過措置で当分3年とされるが、紛争・監査を考えると5年保持が安全

## 6. GMO連携の選定・実装観点

### GMOあおぞら銀行API
- 自社で送金ロジックを持つため柔軟
- API鍵管理、残高管理、Webhook検証、誤送金、障害時リトライ、法務整理を**自社で負う**
- 24/365自動振込に向くが、実装責任は重い

### GMO PG「即給 byGMO」
- 給与前払い業務代行として、デポジット型・立替型・預託型が用意
- リスク: ベンダーロックイン、手数料設計、個人データ委託、障害時責任分界
- 立替型は貸金業該当性の事実関係を契約で確認すべき
- **初期リリースはGMO PGの方がコンプライアンス/運用負担を下げやすい**

## 7. 監査ログ最小要件

### 必須イベント
ログイン成功/失敗、MFA、パスワードリセット、セッション失効、口座登録/変更/検証、勤怠承認/取消、レビュー提出、前払い可能額スナップショット、申請、承認/拒否、GMO送金依頼、Webhook受信、失敗理由マッピング、返金/取消、上限変更、9割/7割ポリシー変更、緊急停止/解除、管理者権限変更

### 必須項目
`actor` / `subject` / `tenant` / `IP` / `端末指紋` / `request_id` / `idempotency_key` / `金額` / `支払期間` / `変更前後のマスク値` / `GMO取引ID` / `結果` / `理由コード` / `ハッシュチェーン値`

**保存**: 最低5年

## 8. レート制限・上限設計

| 対象 | 推奨値 |
|---|---|
| 出金回数 | 成功3回/日、10回/月。失敗含む申請5回/日 |
| 金額（初期） | 1回5万円、1日5万円、月15万円。かつ「承認済み・控除見込後の前払い可能額の90%」以下 |
| 新口座/新端末 | 72時間1万円/日 |
| 口座変更 | 1回/日、2回/月、3回/90日で手動審査。変更後24時間は出金停止 |
| SMS | 送信3回/10分、5回/日。検証5回失敗でロック |
| ログイン | 5回失敗/15分で段階的ロック |
| テナント全体 | 日次送金総額、GMO失敗率、口座変更急増、同一口座への複数人紐付けで自動停止 |

### 出金エラー表示で**渡してはいけない情報**
口座存在有無、名義一致/不一致の詳細、凍結/反社/AML疑い、会社側残高不足、GMO生レスポンス、署名検証失敗、内部リスクスコア、再試行タイミング

### ユーザー向けメッセージ例
> 振込を完了できませんでした。口座情報を確認するか、時間をおいて再申請してください。サポートコード: XXXX

詳細は管理者限定に。
