# System Advisor 機能拡張 引き継ぎ (2026-05-16)

## このセッションのスコープ

`feature/advisor-sql-tool` ブランチで System Advisor に大規模機能追加を実施。

### 完成した機能

1. **execute_sql ツール** — LLMが任意のSELECT文を実行
   - 多層防御: SELECT/WITH限定、危険キーワード拒否、複文禁止、センシティブカラムブロック、LIMIT自動付与、statement_timeout=10s、READ ONLY
   - 表IDを `T-001` 形式で採番、`advisor_chat_tables` に保存
   - 監査ログを `advisor_sql_audit_logs` に記録

2. **SQL承認モーダル** — 実行前に必ずユーザー承認
   - 目的(purpose)表示、SQL本文は折りたたみ
   - 「セッション中はスキップ」チェックでセッション内自動承認
   - 承認状態バナー + 解除リンク (sqlAutoApprove ON時に表示)

3. **add_tables_to_report ツール** — T-XXXをレポートドラフトに追加
   - ボタン押下/チャット指示の両方から呼べる
   - 既存ドラフトがあれば末尾追記、無ければ新規作成

4. **表の共有メニュー** — Sheetsコピー / レポート送信 / URL共有
   - URL共有は `/advisor/t/[token]` 公開ページ (30日有効)
   - `advisor_chat_tables` に share_token/shared_at/shared_until を追加

5. **セッション跨ぎ参照** — 別セッションの表をDBから引いて表示
   - chat-layout の `fetchAndMapSessionTables` で時系列マッピング (C案: 表のcreated_atより前で最新のassistantメッセージに紐付け)
   - お気に入りなしセッションは30日後にcronで削除

6. **get_table ツール** — LLMが過去のT-XXXを参照
   - 「T-005 の合計を出して」→ DB取得 → 集計

7. **GA4 pageTraffic レポート** — ページ×流入元のクロス集計
   - `query_ga4(pageTraffic, page_path_prefix="/lp/30")` のような呼び出し
   - 「LP30 にどこから来た」に対応

8. **Markdown表のT-XXX後付け採番** (`markdown-table-extractor.ts`)
   - assistantメッセージ永続化後、Markdown表を検出してDB登録
   - `**表 T-XXX** (N 行)` プレフィックスを本文に挿入
   - これで GA4/query_metric 等の表も後から get_table / add_tables_to_report / 共有 対象になる

9. **execute_sql 後の loop=1 短絡** — TTFB 100秒問題回避
   - Anthropic API の TTFB 100秒問題対策。execute_sql 成功時はサーバー側固定文で即doneを返す

10. **doneイベントに annotatedContent を含める** — リロード不要でT-XXX反映
   - サーバー側で書き換えた本文を done で返却 → クライアントがstateを更新

11. **システムプロンプト改修**
   - get_table / execute_sql / GA4 pageTraffic の使い方を明示
   - 「取れない理由」の伝え方ルール (個人情報・未実装・データなし)

12. **Search Console (`SEARCH_CONSOLE_SITE_URL`) 環境変数設定** — `.env.local` のみ
   - 現状: `https://tastas.work/` (URLプレフィックスプロパティ用)
   - **ただし認証エラーで動いていない (後述)**

13. **GA4 サービスアカウントキー再発行**
   - `credentials/ga-service-account.json` に新キー配置済み
   - GA4 は動作確認済み

## 残課題

### 🔴 P0: Search Console が認証エラーで動かない

**現象:**
```
Search Console 取得失敗: invalid_grant: Invalid grant: account not found
```

**試したこと:**
1. ✅ `SEARCH_CONSOLE_SITE_URL=sc-domain:tastas.work` (ドメインプロパティ) → 失敗
2. ✅ Search Console の Google Cloud で API 有効化済み (https://console.cloud.google.com/apis/library/searchconsole.googleapis.com?project=tastas-488506)
3. ✅ ドメインプロパティ `tastas.work` の「ユーザーと権限」に `tastas-api-user@tastas-488506.iam.gserviceaccount.com` 追加 → ユーザーは「委任所有者」状態で動かず
4. ✅ URLプレフィックスプロパティ `https://tastas.work/` を新規作成 (確認済み所有者)
5. ✅ `SEARCH_CONSOLE_SITE_URL=https://tastas.work/` に切り替え
6. ✅ URLプレフィックスプロパティの「ユーザーと権限」にもサービスアカウント追加
7. ❌ それでも同じエラー `invalid_grant: account not found`

**残された仮説:**
- (a) **GA4 用と Search Console 用で必要な OAuth scope が違うが、`google.auth.JWT` が正しく動いていない可能性**
- (b) **サービスアカウントキー (`9cda5cc7d4...`) が GCP 側で何らかの理由で無効化** (ただしGA4は動いているので可能性低い)
- (c) **search-console-client.ts の auth 実装 (`new google.auth.JWT({...})`) が `keyFile` 経路でも GA4 と異なる挙動**

**次にやること:**
1. `src/lib/search-console-client.ts` の auth 実装を、ga-client.ts と同じ `googleapis.options({ keyFilename })` 経路に書き換えて試す
2. 認証だけ単独実行する小さなスクリプトを書いて、`JWT.authorize()` の生エラーを観察
3. それでもダメなら新しいサービスアカウントを作って GA4/SC 両方に追加してみる

**コード位置:**
- `src/lib/search-console-client.ts` L19-49 (auth 実装)
- `src/lib/ga-client.ts` L8-31 (比較用)

## 補足: 重要な設計決定

- **データ構造化保存(案D)** ではなく **Markdown再パース(案B)** を採用
  - 理由: 実装量を抑える (~400行 vs 1500-3000行)
  - 副作用: LLMが書く表の表記揺れに弱い、桁区切り数値が文字列で残る
  - 数値計算は LLM が Markdown を読んで計算する設計 (get_table経由)

- **表ID `T-001` 形式** は autoincrement の数値IDをゼロ埋め (`T-${id.padStart(3,'0')}`)
  - 全セッション通しの連番、衝突なし

- **共有URL** は `/advisor/t/[token]` 形式、トークンknowing制、30日有効
  - 既存レポート共有 `/advisor/r/[token]` と同じ設計
